What it does

Task graphs
read like prose.

Task DDL is easy to write and hard to read back — especially with fan-outs, stream conditions, and finalizers. This component parses your pipeline, draws it, and asks an LLM to narrate the flow in execution order.

🗺️
Interactive SVG DAG
Roots highlighted green, finalizers dashed amber, conditional nodes labeled. Click any node to inspect its LLM-generated note.
❄️
Snowflake DDL parser
Handles CREATE TASK, AFTER fan-out/fan-in, SCHEDULE, WHEN SYSTEM$STREAM_HAS_DATA(...), and FINALIZE.
🔗
Bring your own graph
Airflow, Dagster, dbt, homegrown — any orchestrator works if you can shape it as nodes + dependsOn.
📝
Flow narration
One-sentence summary, plain-English walkthrough in execution order, per-node one-liners, and risk observations.
⚠️
Risk observations
Missing finalizers, unconditional heavy tasks, fan-in timing hazards — surfaced without a review meeting.
🧩
Headless mode
Use the usePipelineExplainer hook, parsers, and DagView independently to build your own UI.

Input modes

DDL or JSON.
Your call.

SnowflakeExample.tsx
import { PipelineExplainer } from "pipeline-explainer"

<PipelineExplainer
  llm={{ provider: "anthropic", apiKey: import.meta.env.VITE_ANTHROPIC_KEY }}
  context="Nightly load into the sales mart"
  ddl={`
    CREATE OR REPLACE TASK load_root
      WAREHOUSE = etl_wh
      SCHEDULE = 'USING CRON 0 2 * * * UTC'
      AS CALL start_batch();
    CREATE OR REPLACE TASK load_orders
      AFTER load_root
      WHEN SYSTEM$STREAM_HAS_DATA('orders_stream')
      AS INSERT INTO orders_fact ...;
    CREATE OR REPLACE TASK cleanup
      FINALIZE = load_root
      AS CALL log_batch_end();
  `}
/>
JsonDagExample.tsx
<PipelineExplainer
  llm={{ provider: "openai", apiKey: KEY }}
  pipeline={{
    name: "elt_daily",
    nodes: [
      { id: "extract", dependsOn: [], schedule: "0 1 * * *" },
      { id: "transform", dependsOn: ["extract"] },
      { id: "load", dependsOn: ["transform"] },
    ],
  }}
/>
Heads up: the DDL parser is a pragmatic regex parser for standard task DDL, not a full SQL parser. For unusual DDL, pre-parse yourself and pass a pipeline object. Node bodies are truncated to 500 chars each before being sent to the LLM.

Providers

Any LLM.
One interface.

Switch providers with a single prop. Point at a LiteLLM, vLLM, or Ollama gateway with provider: "custom".

Anthropic
provider: "anthropic"
Default: claude-sonnet-5
OpenAI
provider: "openai"
Default: gpt-4o
Mistral
provider: "mistral"
Default: mistral-large-latest
Custom
provider: "custom"
Any OpenAI-compatible /chat/completions endpoint (LiteLLM, vLLM, Ollama, Azure gateway)

HeadlessUsage.tsx
import {
  parseSnowflakeTasks,
  usePipelineExplainer,
  DagView,
} from "pipeline-explainer"

const pipeline = parseSnowflakeTasks(ddlText)
const { narration, explain } = usePipelineExplainer({
  provider: "custom",
  baseUrl: GATEWAY_URL,
  apiKey: KEY,
  model: "gpt-5-pro",
})
await explain(pipeline, "monthly finance close")

Props

Small surface.
Composable core.

llm: LLMConfig
Required. provider, apiKey, and optionally model, baseUrl, headers.
ddl?: string
Snowflake CREATE TASK DDL script. Provide this or pipeline.
pipeline?: Pipeline
{ nodes: [{ id, dependsOn, schedule?, condition?, finalizer?, body? }] }. Bring any orchestrator's graph.
context?: string
Business context passed to the LLM, e.g. "nightly load into the sales mart".
auto?: boolean
Auto-explain on mount and when the input changes. Defaults to true.
usePipelineExplainer(llm)
Hook that returns { narration, loading, error, explain, reset } for full control over rendering.
